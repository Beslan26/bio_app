# Спецификация проекта: Медицинская платформа на основе ИИ
**Версия:** 1.0.0
**Направление:** Прием пациентов, диагностика и отслеживание состояния здоровья с помощью ИИ


# 1. Системные функции (аутентификация и безопасность)

## 1.1 Аутентификация (на основе JWT)
- **Регистрация**: Регистрация по электронной почте/паролю. Роль по умолчанию: `PATIENT`.

- **Вход в систему**: Проверка учетных данных с поддержкой многофакторной аутентификации (необязательно).

- **Восстановление пароля**: Безопасный процесс на основе токенов (Отправить электронное письмо -> Ссылка для сброса -> Новый пароль).

- **Управление сессиями**:

- `access_token` (кратковременный)

- `refresh_token` (долговременный, хранится в cookie HttpOnly).

- **Безопасность учетной записи**: Возможность смены пароля в рамках активной сессии.

## 1.2 Контроль доступа (RBAC)
Для всех маршрутов API должен быть обеспечен строгий контроль доступа на основе ролей:
- `ADMIN`: Полный доступ к системе, управление пользователями (блокировка/деактивация).
- `DOCTOR`: Доступ к картам пациентов, создание задач, аналитика.
- `PATIENT`: Доступ к собственному профилю, консультации с ИИ, назначенные задачи.

## 1.3 Аудит и соответствие требованиям
- **Журналирование действий**: Каждое изменение состояния (например, обновление статуса, создание задачи) должно регистрироваться в таблице `audit_logs`.
- **Жизненный цикл учетной записи**:
- `active`: Нормальная работа.
- `deactivated`: Приостановка по инициативе пользователя.
- `blocked`: Ограничение по инициативе администратора.

## 1.4 Сущности данных (подсказки по схеме)
- **Пользователь**: `id, email, password_hash, role, status, created_at, last_login`.
- **Журнал аудита**: `id, user_id, action, entity_type, entity_id, timestamp`.

## 1.5 Механизм уведомлений
- **Многоканальная система**:
- `EMAIL`: Критические оповещения (аутентификация, сброс пароля).
- `SMS`: Срочные уведомления от врача или многофакторная аутентификация.
- `SYSTEM`: Уведомления в реальном времени в приложении (Socket/Push).
- **Планировщик**: Использует рабочие/cron задания для отложенных напоминаний (например, «Примите лекарство через 30 минут»).
- **Шаблоны**: Сообщения уведомлений должны быть отделены от логики (хранятся в виде шаблонов).
- **Настройки**: Пользовательские параметры для включения/выключения определенных типов уведомлений.

## 1.6 Управление документами и файлами (УВД)
- **Безопасность прежде всего**: файлы хранятся в изолированных хранилищах (например, S3). Доступ только через подписанные URL-адреса.
- **Логика хранения**:
- `UPLOAD`: поддержка PDF, JPEG, PNG (максимум 10 МБ).
- `ENCRYPTION`: все медицинские записи шифруются в состоянии покоя.
- **Версионирование**:
- Отслеживание `version_id`, `created_by`, `timestamp`.
- Старые версии должны архивироваться, а не удаляться.
- **Матрица разрешений**:
- `PATIENT`: может читать/загружать собственные файлы.
- `DOCTOR`: доступ на чтение к файлам назначенного пациента. Нет прав на удаление.

## 1.7 Анализ данных и экспорт
- **Логика агрегации**: бэкэнд предварительно рассчитывает метрики (например, «Статистика здоровья за определенный период времени») для графиков на фронтенде.
- **Механизм экспорта**:
- `PDF`: для официальных медицинских отчетов/сводок.
- `CSV`: для экспорта необработанных данных (только для пациентов/администраторов).
- **Целостность данных**: исторические снимки пользовательских данных для обеспечения неизменяемости «Истории изменений».

## 1.8 Сущности данных (подсказки по схеме)
- **Уведомление**: `id, user_id, type, status (отправлено/неудачно), payload, scheduled_at`.
- **Документ**: `id, owner_id, file_path, version, type, is_archived`.
- **Журнал аудита**: `id, entity_id, change_data (JSON), updated_by, timestamp`.

# 2. Функции врача (клиническое рабочее пространство)

## 2.1 Медицинская аккредитация и аутентификация
- **Процесс верификции**:
    - Вход в систему с использованием `License Number` + `Credentials`.
    - **Проверка лицензии**: Сервис должен проверить наличие действующих лицензий во внешней/внутренней базе данных.
    - **Статус учетной записи**: Новые профили врачей по умолчанию имеют статус `PENDING_VERIFICATION`. Доступ к данным пациентов ограничен до статуса `VERIFIED`.
- **Связь профиля**: Каждый уникальный идентификатор лицензии должен быть связан с одним `User ID`.

## 2.2 Панель управления врача (рабочее пространство)
- **Управление профилем**:
    - Только для чтения: `License Number`, `Verification Date`.
    - Редактируемые: `Bio`, `Work Hours`, `Contact Info`.
- **Схема специализации**:
    - `Primary Specialization`: (например, кардиология, терапия).
    - `Sub-specializations`: Теги или связанные медицинские поля для системы подбора.
- **Управление пациентами (список пациентов)**:
    - **Активные пациенты**: Пациенты, находящиеся на лечении или имеющие записи на прием.
    - **Очередь/Входящие**: Новые карточки, подготовленные ИИ, ожидающие первичной проверки.
    - **Поиск и фильтр**: Поиск пациентов по имени, тегам симптомов или уровню срочности (назначен ИИ).

## 2.3 Сущности данных (подсказки по схеме)
- **Профиль врача**: `id, user_id, license_id, specialization_id, bio, verification_status (ожидание/проверено/отклонено), verified_at`.
- **Лицензия**: `id, license_number, issue_date, expiry_date, issuer, is_valid (boolean)`.
- **Специализация**: `id, name, description, category`.

## 2.4 Планирование и запись на прием
- **Календарный вид**: Интерфейс для визуализации ежедневных/еженедельных/ежемесячных временных промежутков.
- **Жизненный цикл записи на прием**:
    - `Filters`: Сортировка по дате, времени и приоритету пациента.
    - `Details`: Прямая ссылка на структурированную ИИ жалобу и запись пациента.
    - `Status`: (Запланировано, Завершено, Неявка, Отменено).
- **История**: Неизменяемый архив всех прошлых консультаций для аудита и клинического анализа.

## 2.5 Клиническое управление пациентами (расширенный режим)
- **Полный профиль**:
    - **История болезни**: Хронология предыдущих диагнозов, процедур и сводных данных, полученных с помощью ИИ.
    - **Документация**: Единый обзор результатов лабораторных исследований, изображений DICOM (опционально) и отчетов PDF.
- **Расширенный контекст (социальные детерминанты здоровья)**:
    - **Семейное древо (родственные связи)**: Связь с другими идентификаторами пациентов в системе для отслеживания наследственных рисков.
    - **Окружающая среда**: Записи об условиях жизни и рабочем месте (важно для профессиональных заболеваний/аллергий).
- **Аналитика и тенденции здоровья**:
    - **Визуальное отслеживание**: Интеграция с графиками на интерфейсе (например, артериальное давление, уровень глюкозы с течением времени).
    - **Базовый уровень против текущего**: Выделение значительных отклонений в показателях пациента с помощью ИИ.

## 2.6 Сущности данных (подсказки по схеме)
- **Appointment**: `id, doctor_id, patient_id, start_time, end_time, status, notes`.
- **PatientRelation**: `id, patient_id_1, patient_id_2, relation_type (например, Parent/Child, Spouse)`.
- **PatientSocialData**: `id, patient_id, workplace, residence_type, environment_hazards`.
- **HealthMetric**: `id, patient_id, metric_type, value, unit, timestamp`.

## 2.6 Диагностика и клинические записи
- **Предварительный диагноз**: Интерфейс для ввода врачом первоначального диагноза (с привязкой к кодам МКБ при необходимости). Это поле отличается от предложения ИИ.
- **Примечания к консультации**: Поддержка форматированного текста для подробных медицинских записей во время/после приема.
- **Комментарий к лабораторным исследованиям**: Возможность прикреплять конкретные примечания/интерпретации к загруженным медицинским документам или результатам лабораторных исследований.
- **Рекомендации**: Отформатированный блок инструкций для пациента (отдельно от отслеживаемых задач).

## 2.7 Управление задачами (отслеживание пациентов)
- **Механизм создания задач**:
    - **Типы**:
        - `Numeric`: Для отслеживания показателей (например, артериальное давление, вес, температура).
        - `Text/Journal`: Для качественной обратной связи (например, «Опишите свое настроение» или «Перечислите свои приемы пищи»).
        - `Checkbox`: Для простого заполнения (например, «Принимать лекарства»).
- **Планирование**: Определение дат начала/окончания и частоты (ежедневно, еженедельно, с определенными интервалами).
- **Мониторинг и обратная связь**:
    - **Панель мониторинга выполнения**: Визуальный индикатор соблюдения пациентом режима лечения (процент выполненных задач).
    - **Анализ**: Агрегирование числовых данных в линии тренда.
- **Жизненный цикл**:
- `Active`: Отслеживается в данный момент.
- `Completed`: Задача завершена.
- `Extended`: Обновление периода для существующей задачи.
- `Closed/Archived`: Остановлено врачом.

## 2.8 Сущности данных (подсказки по схеме)
- **Diagnosis**: `id, doctor_id, patient_id, diagnosis_text, icd_code, type (preliminary/final)`.
- **PatientTask**: `id, doctor_id, patient_id, title, task_type (numeric/text/boolean), unit (if numeric), frequency, start_date, end_date, status`.
- **TaskEntry**: `id, task_id, value (string/number), timestamp, patient_comment`.

## 2.9 Коммуникации и аналитика ИИ
- **Входящие жалобы**:
    - Уведомления о новых заявках от пациентов в режиме реального времени.
    - Просмотр исходных данных от пациентов (текст/аудиозапись) вместе со структурированными данными ИИ.
- **Интерфейс сводки ИИ**:
    - Визуальное выделение ключевых симптомов, маркеров срочности и потенциальных тревожных сигналов, обнаруженных ИИ.
    - Функция «Принять сводку ИИ» одним щелчком мыши для импорта данных в официальную медицинскую карту.
- **Система обмена сообщениями**:
    - **Защищенный чат**: текстовая коммуникация в контексте «пациент-врач».
    - **Контекстные ответы**: возможность отправлять сообщения, связанные с конкретными задачами или результатами лабораторных исследований.
    - **Уведомления о прочтении**: отслеживание статуса доставки сообщений пациентам.

## 2.10 Сущности данных (подсказки по схеме)
- **CommunicationThread**: `id, doctor_id, patient_id, created_at, last_message_at`.
- **ChatMessage**: `id, thread_id, sender_id, role (doctor/patient), content, is_read, metadata (link to task/file)`.
- **ComplaintRecord**: `id, patient_id, raw_input, ai_summary_json, urgency_score, status (new/reviewed/archived)`.

# 3. Функции для пациентов (пользовательский опыт)

## 3.1 Профиль и соответствие требованиям
- **Личные данные**: полное имя, дата рождения, пол, группа крови.
- **Динамические обновления**: возможность редактирования контактных данных и контактной информации для экстренной связи.
- **Соответствие законодательству (GDPR/FZ)**:
    - **Управление согласием**: явные переключатели для обработки персональных данных и хранения медицинских данных.
    - **Журнал аудита**: запись о том, когда и какая версия согласия была подписана.
    - **Удаление данных**: запрос на деактивацию учетной записи и обфускацию данных.

## 3.2 Подача жалоб (с использованием ИИ)
- **Многомодальный ввод**:
    - **Текст**: Стандартное текстовое поле для описания симптомов.
    - **Голос (STT)**: Функция аудиозаписи с автоматическим преобразованием речи в текст.
- **История поданных жалоб**: Панель мониторинга, отображающая все прошлые жалобы с их статусом (Обрабатывается, Рассмотрено врачом, Архивировано).
- **Сортировка с помощью ИИ (в режиме реального времени)**:
    - **Извлечение симптомов**: ИИ идентифицирует ключевые слова и закономерности во входных данных.
    - **Оценка срочности**: Мгновенная обратная связь о тяжести состояния (например, «Обратиться за неотложной помощью» против «Дождаться специалиста»).
- **Подбор специалиста**:
- Логика для рекомендации конкретных медицинских областей на основе симптомов, проанализированных ИИ (например, «На основании ваших симптомов мы рекомендуем кардиолога»).

## 3.3 Сущности данных (подсказки по схеме)
- **PatientProfile**: `id, user_id, full_name, dob, gender, blood_type, emergency_contact`.
- **Consent**: `id, user_id, consent_type, version, signed_at, ip_address`.
- **SymptomSubmission**: `id, patient_id, input_type (text/audio), raw_content, audio_url, transcription, ai_results_json`.

## 3.4 Персональная медицинская карта (PHR)
- **Медицинская панель**:
    - **Хронология здоровья**: Централизованный обзор всех прошлых диагнозов и медицинских событий.
    - **Хранилище документов**: Личное хранилище результатов анализов, рецептов и изображений.
- **Советы врача**: Специальный раздел для рекомендаций, которые нельзя отслеживать (например, «Общие рекомендации по образу жизни» от врача).
- **Записи только для чтения**: Диагнозы и официальная история болезни должны быть неизменяемыми (пациент не может редактировать официальные записи врача).

## 3.5 Выполнение задач (отслеживание состояния пациента)
- **Активный список задач**: Целенаправленный просмотр «Что нужно сделать сегодня» (флажки, поля ввода).
- **Логика ввода данных**:
    - **Проверка**: Убедитесь, что числовые значения (например, артериальное давление) находятся в пределах реалистичных диапазонов.
    - **Редактирование**: Разрешите редактирование записей только в течение определенного периода времени (например, «В течение 24 часов с момента ввода»).
- **Прогресс и визуализация**:
    - **Геймификация/Последовательность**: Визуальная полоса прогресса или процент выполнения для поощрения соблюдения рекомендаций.
    - **Персональная аналитика**: Интерактивные диаграммы, отображающие собственные данные пациента (пульс, вес и т. д.) для визуализации процесса выздоровления.

## 3.6 Сущности данных (подсказки по схеме)
- **PatientDocument**: `id, patient_id, title, file_url, uploaded_at, category (Lab/Scan/Prescription)`.
- **TaskEntry**: `id, task_id, value, patient_comment, timestamp, last_edited_at`.
- **HealthSnapshot**: `id, patient_id, key_metrics_json, generated_at` (для быстрой загрузки диаграмм на панели мониторинга).

## 3.7 Оповещения и уведомления для пациентов
- **Напоминания о задачах**:
    - Триггеры push-уведомлений/отправки электронных писем на основе полей `frequency` и `scheduled_at` назначенных задач.
    - Логика эскалации: «Напоминание отправлено за 15 минут до» и «Оповещение о просроченной задаче», если задача не выполнена.
- **Оповещения о записи на прием**:
    - Стандартные уведомления: за 24 часа, за 1 час и «Кабинет готов» (для телемедицины).
- **Прямые обновления от врача**:
    - Мгновенные оповещения, когда врач:
        - Обновляет диагноз.
        - Комментирует результаты анализов.
        - Отправляет новое сообщение в чат.
        - Закрывает или продлевает задачу.

## 3.8 Сущности данных (подсказки схемы)
- **NotificationLog**: `id, user_id, trigger_event (task_due/appt_soon/doc_message), channel (push/email), status (pending/sent/read)`.
- **UserPreferences**: `id, user_id, enable_email, enable_push, quiet_hours_start, quiet_hours_end`.

# 4. ИИ-движок (умный помощник)

## 4.1 Преобразование речи в медицинский текст (STT)
- **Транскрипция**: Преобразование аудиозаписи голоса пациента в необработанный текст (с использованием Whisper или аналогичной программы).
- **Шумоподавление и очистка**: ИИ должен удалить слова-паразиты («э-э», «ну» и т. д.) и нерелевантную «шумовую» информацию, сохраняя при этом клинический контекст.

## 4.2 Извлечение клинических сущностей (NLP)
Искусственный интеллект должен проанализировать очищенный текст и извлечь следующие сущности в структурированный JSON:
- **Симптомы**: Список выявленных проблем со здоровьем.
- **Длительность/Начало**: Когда началось (например, «3 дня назад»).
- **Интенсивность**: Шкала боли/дискомфорта (1-10) или качественная оценка (например, «острая», «тупая»).
- **Факторы риска**: Упоминания об аллергии, хронических заболеваниях или генетической предрасположенности.
- **Локализация**: Где именно болит.

## 4.3 Структурирование и суммирование данных
- **Карточка жалобы**: Структурированный JSON-объект для бэкэнда системы.
- **Медицинское заключение (мнение врача)**: Профессиональное, краткое изложение с использованием медицинской терминологии (например, «Пациент сообщает о пароксизмальной тахикардии в течение 48 часов»).
- **Краткое изложение состояния пациента**: Упрощенная версия для подтверждения пациентом («Вы сообщили нам о сердцебиении, которое началось 2 дня назад»).

## 4.4 Сортировка и маршрутизация
- **Подбор специалистов**:
    - ИИ анализирует извлеченные симптомы по базе данных «Специализации».
    - Результат: 3 наиболее рекомендуемых типа врачей.
- **Оценка срочности (сортировка)**:
    - `КРАСНЫЙ`: Экстренная ситуация (боль в груди, симптомы инсульта) -> Тревога для немедленных действий.
    - `ЖЕЛТЫЙ`: Требуется внимание в течение 24 часов.
    - `ЗЕЛЕНЫЙ`: Плановая консультация.
- **Предупреждения о риске**: Автоматическое обнаружение «красных флагов» (например, определенные комбинации симптомов, требующие неотложной помощи).

## 4.5 Сущности данных (подсказки схемы)
- **Задание обработки ИИ**: `id, submission_id, status (processing/completed/failed), model_version, tokens_used`.
- **StructuredComplaint**: `id, submission_id, entities (JSON: symptoms, duration, intensity), triage_level, recommended_spec_id`.

# 6. Сущности и связи предметной области (модель данных)

В этом разделе определены основные сущности. ИИ должен использовать это в качестве основы для схемы базы данных (Prisma/SQL).

## 6.1 Основные сущности пользователя
- **Пользователь**: Базовая идентификация (электронная почта, авторизация, глобальная роль).
- **Врач**: Расширенный профиль (лицензия, биография, статус подтверждения). Связан один к одному с пользователем.
- **Пациент**: Личный медицинский профиль (дата рождения, пол, социальные данные). Связан один к одному с пользователем.

## 6.2 Медицинские записи
- **Медицинская карта (PHR)**: Основной контейнер для истории болезни пациента.
- **Медицинская карта**: Отдельное клиническое событие (диагноз, клиническая заметка). Связана с врачом и пациентом.
- **Исследование (анализ/сканирование)**: Результаты лабораторных исследований или файлы изображений. Может быть связана с медицинской картой.

## 6.3 Процесс консультации
- **Жалоба**: Первоначальный ввод данных пациентом + JSON-данные, структурированные с помощью ИИ.
- **Приём (Приём)**: Запланированная или прошедшая встреча. Связывает жалобу, врача и пациента.
- **Рецепт (Назначение)**: Официальные медицинские назначения (лекарства, процедуры).

## 6.4 Отслеживание и взаимодействие
- **Задача пациента**: Трекер, созданный врачом (числовой/текстовый/флажок).
- **Результат задачи**: Ежедневные/периодические записи, сделанные пациентом для конкретной задачи.
- **Уведомление**: Оповещения, срабатывающие при системных событиях (запланированные задачи, новые сообщения).

## 6.5 Карта сущностей и связей (ERM)
- **Пользователь** -> (имеет один) -> **Врач ИЛИ Пациент**.
- **Пациент** -> (имеет много) -> **Жалобы** и **Медицинские записи**.
- **Врач** -> (управляет многими) -> **Записи на прием** и **Задачи**.
- **Задача** -> (содержит много) -> **Результаты задачи**.
- **Медицинская карта** -> (может содержать много) -> **Исследования**.

- **Логика**: Независимое логирование всех операций, связанных с безопасностью.
- **Соответствие требованиям**: Обеспечивает неизменяемость и возможность поиска журналов для административного анализа.

# 8. Поток взаимодействия пользователя: Цикл взаимодействия пациента и врача

В этом разделе определяются последовательные состояния системы. Переходы между шагами должны запускать определенные обновления пользовательского интерфейса и изменения доступа к данным.

## 8.1 Основные этапы потока
1. **Ввод данных**: Пациент сообщает о симптомах (текст/голос) -> Создается `Заявка` (Статус: `ОЖИДАЕТСЯ_AI`).
2. **Обработка**: Сервис ИИ извлекает сущности -> Создается «Карточка жалобы» (Статус: `ГОТОВА_К_СОПОСТАВЛЕНИЮ`).
3. **Маршрутизация**: Система определяет специальность -> Назначает врачу (Статус: `ОЖИДАЕТСЯ_ПРОВЕРКИ`).
4. **Предварительная консультация**: Врач открывает `Карточку жалобы` (только для чтения сводка ИИ).
5. **Консультация**: Врач инициирует «Активную сессию» -> Прямое общение/осмотр (Статус: `В ПРОЦЕССЕ`).
6. **Назначение лекарств**: Врач создает `Задачи пациента` (трекеры) -> Пациенту предоставляется доступ.
7. **Соблюдение режима лечения**: Пациент вносит данные в трекеры -> Записи задач заполняются в режиме реального времени.
8. **Клинический обзор**: Врач просматривает сводные карты/отчеты -> Принимает решение о `Завершении`, `Продлении` или `Корректировке` лечения.

## 8.2 Логика перехода состояний (для ИИ-агента)
- **Контроль доступа**: Шаг 6 (Назначение задачи) должен быть заблокирован до тех пор, пока врач не инициирует Шаг 5 (Консультация).
- **Доступность данных**: Шаг 8 (Анализ) требует как минимум одной записи задачи от пациента для создания отчета.
- **Уведомления**: Автоматические оповещения должны срабатывать на шаге 3 (для врача) и шаге 6 (для пациента).
